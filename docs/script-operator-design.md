# auto_wine 脚本操作器正式设计文档

> 文档版本：v0.1  
> 适用项目：`auto_wine`  
> 定位：Fabric 客户端模组内的脚本式自动化编排与执行系统

## 1. 文档目的

本文档用于定义 `auto_wine` 中“脚本操作器”的正式定位、系统边界、用户流程、核心模块、数据模型、交互形态以及落地路线。

脚本操作器不是单纯的按键宏，也不是只负责记录操作的录像器，而是一个面向 Minecraft 客户端场景的**脚本化执行框架**。它的目标是把“在什么界面、拿什么物品、执行什么动作、按什么顺序、失败后如何处理”抽象成可编辑、可保存、可复用、可导入导出的脚本。

### 1.1 文档维护原则

- 先维护总纲，再补实现细节，避免文档早于代码定型
- 优先描述稳定概念，如术语、边界、流程和模块职责
- 具体字段、接口和数据结构，尽量等对应功能进入实现阶段再展开
- 文档内容应与 `plan.md` 保持同步，阶段任务变化时同步修订
- 如果后续实现和设计有偏差，优先更新文档中的约束与说明

## 2. 脚本操作器的定位

### 2.1 核心定位

脚本操作器是 `auto_wine` 的**执行层**与**编排层**，负责将用户定义的流程转化为客户端可执行的操作序列。

它的职责包括：

- 根据当前界面和上下文识别脚本是否可执行
- 根据脚本定义生成动作序列
- 在客户端侧执行点击、拖拽、放置、收取、切换手持物等操作
- 在 HUD 或引导模式中展示当前执行状态
- 处理暂停、中止、失败恢复和安全校验

### 2.2 不做什么

脚本操作器**不**承担以下职责：

- 不直接替代服务端逻辑
- 不改变 Minecraft 原生规则
- 不负责底层物品识别算法本身的细节实现
- 不把所有配置逻辑硬编码到脚本中
- 不将自动化行为无条件开放给所有场景

### 2.3 与现有框架的关系

脚本操作器应建立在当前工程已有基础上：

- `AutoWineFramework`：公共初始化入口
- `AutoWineClientFramework`：客户端初始化入口
- `feature`：功能模块注册与生命周期管理
- `config`：功能开关、风控配置、调试配置
- `storage`：脚本、缓存、运行记录、用户数据持久化

也就是说，脚本操作器应作为一个或多个 `feature` 模块逐步接入，而不是脱离现有架构单独编写。

## 3. 设计目标

### 3.1 功能目标

1. 支持“一个脚本描述一条操作流程”
2. 支持脚本编辑、保存、加载、导出、导入
3. 支持脚本复用，即脚本中调用脚本片段或子流程
4. 支持图形化编辑与执行引导
5. 支持运行态高亮与 HUD 提示
6. 支持不记录箱子位置、依靠信息记录自动定位目标物品
7. 支持基于上下文的最小闭环自动执行

### 3.2 体验目标

- 对用户来说，脚本应像“流程图 + 操作清单”
- 用户可以从“记录”开始，也可以从“手工编辑”开始
- 执行时既可以全自动，也可以半自动引导
- 所有自动化行为都应可暂停、可停止、可回退

### 3.3 工程目标

- 数据结构清晰，便于扩展
- 脚本格式稳定，便于版本迁移
- 执行引擎与 UI 层解耦
- 风控逻辑独立，避免误操作
- 后续可扩展到更多容器、更多物品与更多动作类型

## 4. 概念定义

### 4.1 脚本（Script）

脚本是一个可序列化的流程定义，描述：

- 前置条件
- 目标对象
- 操作步骤
- 子脚本引用
- 失败处理方式
- 执行结束条件

### 4.2 步骤（Step）

步骤是脚本的最小执行单元，通常包含：

- 动作类型
- 目标位置或目标物品
- 操作参数
- 依赖条件
- 执行结果判断

### 4.3 动作（Action）

动作是客户端执行器真正发出的交互行为，例如：

- 打开容器
- 读取槽位
- 点击格子
- 拖拽物品
- 将物品放入左手
- 右键使用
- 收取结果

### 4.4 上下文（Context）

上下文是脚本执行时的环境信息，包括：

- 当前打开的界面
- 当前玩家背包状态
- 当前容器快照
- 当前选中物品
- 当前脚本执行进度
- 当前风险状态

### 4.5 子脚本 / 函数（Subscript / Function）

子脚本是可以被复用的流程片段，相当于脚本中的函数。

例如：

- “把发光浆果放到左手”
- “执行一次混合动作”
- “将物品放入指定容器”

主脚本可以调用这些子脚本，从而减少重复编辑。

## 5. 用户流程设计

### 5.1 总体用户流程

脚本操作器的标准使用流程建议分为六步：

1. **创建脚本**
   - 新建空脚本
   - 从模板生成脚本
   - 从录制结果生成脚本
   - 导入外部脚本

2. **定义流程**
   - 选择目标界面或目标场景
   - 添加步骤
   - 调整步骤顺序
   - 配置每一步的参数和约束

3. **校验脚本**
   - 校验脚本语法
   - 校验引用的子脚本是否存在
   - 校验动作是否匹配当前场景
   - 校验风险规则是否允许执行

4. **预览与引导**
   - 在编辑器中预览脚本流程图
   - 在 HUD 中显示下一步
   - 在引导模式中按步骤提示执行

5. **执行脚本**
   - 进入自动执行或半自动执行
   - 根据上下文动态推进步骤
   - 记录执行日志和异常信息

6. **保存与复用**
   - 自动保存编辑结果
   - 支持导出脚本文件
   - 支持作为子脚本复用

### 5.2 三种典型使用方式

#### A. 手工编排

用户在编辑器里直接构建脚本：

- 选择目标场景
- 添加“左手持物品”步骤
- 添加“右键放入”步骤
- 添加“执行混合脚本”步骤
- 保存为可复用脚本

适合：熟悉流程、追求稳定、长期维护的用户。

#### B. 录制生成

用户点击“开始录制”，系统记录操作行为，自动生成脚本草稿。

适合：希望快速建立初稿、再人工微调的用户。

#### C. 引导执行

用户不直接让系统替自己操作，而是通过 HUD 查看下一步动作，并手动确认执行。

适合：新手、测试阶段、风险较高的环境。

## 6. 系统流程设计

### 6.1 执行管线

脚本操作器建议采用以下执行管线：

**感知 → 解析 → 规划 → 执行 → 反馈**

#### 1. 感知

采集当前游戏状态：

- 优先通过**容器预览**采集容器内容与槽位信息
- 容器内容
- 背包内容
- 当前屏幕类型
- 玩家持有物品
- 容器位置或识别信息

#### 2. 解析

把脚本和当前上下文进行对比：

- 当前脚本是否适配当前场景
- 是否满足前置条件
- 是否需要先获取物品
- 是否存在缺料或冲突

#### 3. 规划

将脚本步骤转成可执行队列：

- 识别步骤顺序
- 解析子脚本引用
- 展开条件分支
- 生成动作序列

#### 4. 执行

通过客户端交互层执行动作：

- 点击
- 拖拽
- 放置
- 取回
- 切换快捷栏
- 右键交互

#### 5. 反馈

根据执行结果更新状态：

- 步骤成功/失败
- 当前进度
- 错误原因
- 是否进入暂停或中止

### 6.2 状态机建议

脚本执行应采用显式状态机，避免“黑盒式连续运行”。推荐状态如下：

- `IDLE`：空闲
- `READY`：已加载脚本，等待条件满足
- `RUNNING`：执行中
- `PAUSED`：暂停
- `WAITING_CONFIRMATION`：等待用户确认
- `FAILED`：执行失败
- `STOPPED`：用户中止
- `COMPLETED`：执行完成

## 7. 核心模块拆分

### 7.1 推荐模块划分

建议按功能拆分为以下子模块：

- `script`：脚本模型、解析、执行、校验
- `script.editor`：脚本编辑器
- `script.hud`：HUD 显示与状态提示
- `script.guide`：引导模式
- `script.io`：导入导出、版本迁移
- `script.runtime`：执行上下文与状态机
- `script.safety`：限速、场景校验、终止条件
- `script.registry`：脚本模板与子脚本注册

### 7.2 与现有功能模块的关系

脚本操作器未来应与以下能力协作：

- **容器扫描**：提供可执行的物品与槽位信息
- **配方匹配**：把当前物品组合映射到目标流程
- **自动化操作**：将动作序列落地为客户端交互
- **配置系统**：控制启用范围和安全策略
- **存储系统**：保存脚本、模板、记录与执行日志

## 8. 数据模型建议

### 8.1 脚本元信息

脚本建议至少包含：

- `id`：脚本唯一标识
- `name`：脚本名称
- `description`：脚本说明
- `version`：脚本格式版本
- `author`：作者
- `tags`：标签
- `createdAt` / `updatedAt`：时间戳
- `enabled`：是否启用

### 8.2 脚本主体

脚本主体建议包含：

- `scope`：适用场景
- `steps`：步骤列表
- `subscripts`：子脚本引用
- `conditions`：前置条件
- `fallbacks`：失败处理
- `metadata`：扩展字段

### 8.3 步骤结构

每个步骤建议包含：

- `type`：动作类型
- `target`：目标对象
- `input`：输入参数
- `output`：结果判断
- `constraints`：约束条件
- `timeout`：超时限制
- `onFail`：失败策略

### 8.4 运行时结构

运行时建议保存：

- 当前脚本 id
- 当前步骤索引
- 当前状态
- 当前界面信息
- 当前容器快照
- 当前错误信息
- 最近一次执行日志

### 8.5 持久化建议

持久化分两类：

1. **配置类**：开关、风控规则、界面偏好
   - 适合放在 `config`
2. **内容类**：脚本、模板、录制结果、运行记录
   - 适合放在 `storage`

## 9. 编辑器、HUD 与引导模式

### 9.1 编辑器

编辑器是脚本操作器的主要生产界面。

#### 目标

- 以流程图或步骤列表方式编辑脚本
- 支持拖拽排序、分组、折叠
- 支持查看每个步骤的参数和依赖
- 支持局部复用子脚本
- 支持即时校验和错误标记

#### 推荐交互

- 左侧：脚本结构树 / 流程图
- 中间：当前步骤编辑区
- 右侧：属性面板 / 约束面板
- 底部：日志 / 校验结果 / 录制控制

### 9.2 HUD 模式

HUD 是运行时提示层，用于展示脚本执行状态。

#### 应显示内容

- 当前脚本名称
- 当前步骤进度
- 下一步动作
- 风险警告
- 缺料提示
- 暂停 / 停止状态

#### 适用场景

- 自动执行时显示进度
- 半自动执行时显示下一步
- 调试时显示执行日志摘要

### 9.3 引导模式

引导模式是半自动执行方式，不替玩家完成动作，只负责提示和引导。

#### 目标

- 降低新手使用门槛
- 提供流程教学
- 在风险较高时作为保守执行方案

#### 行为

- 不主动点击
- 不主动拖拽物品
- 只提示“当前应该做什么”
- 可在每步确认后继续

## 10. 脚本复用与“脚本中使用脚本”

脚本复用是脚本操作器的关键能力之一。

### 10.1 复用方式

建议支持两种复用方式：

1. **子脚本调用**
   - 主脚本在某一步引用某个子脚本
   - 执行时自动展开为子流程

2. **模板实例化**
   - 通过模板生成相似脚本
   - 例如替换目标物品、替换容器位置、替换操作次数

### 10.2 示例语义

例如：

- 选择发光浆果
- 放到左手
- 执行“混合脚本”
- 再放入目标容器

其中“混合脚本”就是可复用的子脚本。

### 10.3 复用原则

- 子脚本尽量无副作用或副作用可控
- 子脚本参数应明确
- 子脚本执行失败要能回传到主脚本
- 复用层级不宜过深，防止调试困难

## 11. 导入导出设计

### 11.1 导出目标

脚本需要支持导出，方便：

- 备份
- 分享
- 跨设备迁移
- 多版本管理

### 11.2 建议格式

推荐以 **JSON** 作为主导出格式，因为它：

- 易于程序读取
- 易于版本升级
- 便于调试和差异比较

如后续需要，也可以附加人类可读的展示格式，但核心数据建议保持 JSON。

### 11.3 版本迁移

导入时应进行：

- 版本检查
- 缺失字段补全
- 旧字段映射
- 不兼容脚本的降级处理

### 11.4 位置无关性

用户提到“位置不同，要修改”，这意味着脚本不应把容器位置写死为唯一依赖。

建议脚本支持两类定位方式：

1. **位置绑定**：明确指定坐标、容器位置
2. **信息绑定**：通过容器内容、命名、类型、上下文自动识别

实际执行时，应优先尝试信息绑定，并以**容器预览**作为物品读取的首选来源；必要时再使用位置绑定作为补充。

---

## 12. 信息记录与脚本生成

### 12.1 手动信息记录

用户可手动记录：

- 打开了什么箱子
- 箱子里有什么物品（可由容器预览自动读取，手动记录用于补全和校验）
- 每个物品的数量
- 每个物品所在位置

这类记录可作为脚本上下文和容器画像；当容器预览不完整、不可用或需要人工确认时，手动记录作为补充数据源和兜底方案。

### 12.2 配方录制

用户开始录制后，系统记录以下信息：

- 玩家执行了哪些交互
- 交互顺序是什么
- 哪一步对应哪种操作
- 哪些步骤可以抽象成子脚本

最终生成一个可编辑脚本草稿。

### 12.3 外部配方导入

脚本操作器应该支持导入外部定义的配方或流程模板，并将其转换为本地脚本模型。

---

## 13. 安全边界与风控

脚本操作器的设计必须优先考虑安全边界。

### 13.1 默认原则

- 默认只在客户端执行
- 默认支持用户确认
- 默认允许暂停和停止
- 默认禁止在不安全场景中自动化

### 13.2 风控建议

- 自动化总开关
- 速率限制与随机延迟
- 当前界面校验
- 焦点丢失自动停机
- 失败次数上限
- 高风险界面只读预览

### 13.3 异常处理

遇到以下情况应停止或降级：

- 当前界面与脚本预期不符
- 容器内容变化过大
- 关键物品缺失
- 执行超时
- 用户手动中止
- 客户端焦点丢失

## 14. 与现有阶段规划的关系

脚本操作器不是一开始就完整实现，而应建立在当前阶段规划之上。

### 14.1 当前优先级

当前最优先的底层能力仍然是：

1. 物品 / 容器扫描
2. 配方数据模型
3. 配方匹配引擎
4. 自动化操作
5. 安全保护与测试

### 14.2 脚本操作器的落点

脚本操作器会贯穿以上五项能力：

- 扫描层提供“看见什么”
- 数据模型层提供“如何描述流程”
- 匹配层提供“现在适合执行什么”
- 自动化层提供“如何真正执行”
- 安全层提供“什么时候不能执行”

---

## 15. 推荐落地路线

### 阶段 1：脚本模型与存储

目标：先把脚本定义下来。

- 定义脚本、步骤、子脚本、运行状态的数据模型
- 完成序列化与反序列化
- 接入 `storage`
- 支持保存、加载、删除、重命名

### 阶段 2：最小编辑器

目标：先能编辑脚本。

- 提供步骤列表编辑
- 提供基础属性面板
- 支持新增/删除/排序步骤
- 支持脚本校验

### 阶段 3：HUD 预览与引导模式

目标：先能看见执行进度。

- 显示当前步骤
- 显示下一步动作
- 显示风险提示
- 支持手动确认推进

### 阶段 4：执行引擎接入

目标：让脚本真正动起来。

- 把脚本步骤转换为动作序列
- 接入客户端交互执行
- 支持暂停/继续/停止
- 记录执行日志

### 阶段 5：导入导出与模板复用

目标：让脚本可共享、可复用。

- 支持 JSON 导出/导入
- 支持脚本模板
- 支持子脚本复用
- 支持版本迁移

### 阶段 6：安全与测试

目标：让系统可长期维护。

- 完善安全边界
- 增加异常处理
- 为核心数据结构补测试
- 为执行流程补回归验证

---

## 16. 建议的首次实现切口

如果要开始编码，建议第一步不是直接做自动点击，而是先落地下面这个最小闭环：

1. 定义脚本数据模型
2. 定义步骤和子脚本引用
3. 完成脚本存储与导入导出
4. 做一个最简编辑器入口
5. 再逐步接入 HUD 和执行引擎

这样做的好处是：

- 后续所有功能都能围绕统一脚本格式展开
- 不会把 UI、执行和数据结构绑死
- 方便先做“半自动引导”再过渡到“自动执行”

## 17. 决策运行时状态机（DecisionRuntime）

`DecisionRuntime` 是连接"本地决策层"与"执行层（如 Baritone）"的核心组件，
采用显式三状态有限状态机保证执行层的可用性不成为系统单点故障。

### 17.1 状态定义

| 状态 | 含义 |
|------|------|
| `ACTIVE` | 执行层（Baritone）健在，接受并转发计划 |
| `DEGRADED` | 执行层已故障，仅运行扫描 + 本地决策；后台每 200 tick 探测一次 |
| `PENDING_RESUME` | 探测成功，等待当前 goal 到达任务边界再切回 ACTIVE |

### 17.2 状态转换规则

```
ACTIVE ──(ExecutionLayer.execute 抛出异常)──> DEGRADED
DEGRADED ──(探测成功 && 当前 goal 在任务边界)──> ACTIVE
DEGRADED ──(探测成功 && 当前 goal 不在任务边界)──> PENDING_RESUME
PENDING_RESUME ──(goal.isAtBoundary() == true)──> ACTIVE
```

### 17.3 关键约定

- **首次异常即降级**：`execute()` 抛出任何异常立即触发降级，不重试。
- **降级后持续运行**：DEGRADED 模式下仍执行 `DecisionPlanner.plan()`，本地决策不中断。
- **后台探测**：每 200 tick（约 10 秒）调用 `ExecutionLayer.isAvailable()`；
  探测成功时进入 `PENDING_RESUME`，不立即切换。
- **任务边界切换**：PENDING_RESUME 状态下，只有当前 goal 的
  `isCompleted() || isFailed() || isStopped()` 为 `true` 时，才切回 ACTIVE，
  避免执行层在任务中途抖动。
- **日志审计**：每次状态切换均在 `AutoWineFramework.LOGGER` 记录，便于排查。

### 17.4 ExecutionLayer 接口约定

实现 `ExecutionLayer` 的组件（如 Baritone 绑定层）须遵守：

- `isAvailable()`：返回执行层是否已加载且就绪；不得抛出异常。
- `execute(plan, context)`：提交计划执行；遇到不可恢复故障时抛出 `RuntimeException`。
- `interrupt()`：中断当前执行；**不得抛出异常**。
- `reset()`：重置内部状态为初始就绪状态；**不得抛出异常**。

### 17.5 与脚本操作器的关系

`DecisionRuntime` 位于执行管线的"执行"环节：

```
感知（ContainerScanFeature）
  → 决策（DecisionPlanner + DecisionGoal）
  → 执行（ExecutionLayer / DEGRADED 本地执行）
  → 反馈（日志 / HUD，待实现）
```

DEGRADED 模式下，感知与决策层不受影响；仅执行层降级为"计划生成但不下发"。
这确保配方匹配、容器扫描等阶段 D 的功能在无 Baritone 时也能正常开发和测试。

## 18. 结论

脚本操作器在 `auto_wine` 中应被定义为：

> 一个面向 Minecraft 客户端场景的、可编辑、可复用、可导入导出、可引导、可暂停的脚本执行系统。

它的正确实现路径不是先做“自动点击”，而是先完成“脚本模型 → 编辑器 → 预览/HUD → 执行引擎 → 安全控制”的分层建设。

只要这条路线走稳，后续无论是容器扫描、配方匹配，还是复杂的操作编排，都会自然接入到同一个系统里。
